**Now known that wasn't.** (1) **biome auto-fixes can land post-commit via the file watcher**, requiring follow-up `chore(format)` commits. PR C and PR E both saw this — the pre-commit hook stages safe auto-fixes into the index before commit, but unsafe-fix suggestions (template literals, optional-chain replacements over non-null assertions, line collapses under 100-char limit) get applied AFTER the commit by the linter watcher running outside the hook surface. Fix forward via separate `chore(format)` commit; **don't try to amend** — amending after auto-fix can mask the lint origin and breaks the pre-commit gate's idempotency claim. (2) **Branch protection's `BEHIND` mergeStateStatus rejects merges even with passing checks** because GitHub re-validates required-check status against the merge-base with the *current* main, not the PR's recorded checks. Solution: merge `origin/main` into the PR branch first (creates a merge commit, no force-push), CI re-runs against the merged tree, then squash-merge. This is the expected workflow for parallel PRs — not a bug. The kickoff's "PR ordering on slip" had this in mind, but the explicit workflow ("update branch with main, wait for fresh CI, merge") didn't surface clearly until I hit the rejection. (3) **Force-push is denied by harness policy even with `--force-with-lease`**, on top of GitHub branch-protection enforcement. Originally rebased `day-6/firehose` onto current main locally to pick up PR #29's revocation infrastructure; pushing the rebased branch failed (different SHA than remote's spec commit). Solution that worked: create a fresh branch `day-7/firehose` from current HEAD and push that — the commit history retained both the original spec commit and the new implementation commit, so PR #30's design-spec content was preserved + #36 superseded it cleanly. (4) **The verification-committer pattern is the natural extension of the revocation-committer pattern.** ADR 0004 addendum-1 + addendum-2 set up the abstraction; PR D applied it to verify-gist with zero conceptual debt. The handler is now purely validation + leaf construction, the committer holds the tx body where ADR 0004 hygiene rules are auditable, and tests inject a `vi.fn` fake instead of mocking Drizzle's transaction surface. This generalizes — any future "X writes happen atomically with `merkle.append` + firehose insert + state transition" handler will use the same shape (`createXCommitter(db, merkle)` factory + `XCommitter` deps slot + handler delegating). (5) **DoH discriminated-status discipline survives translation from Gist.** The 8-way DNS adapter mirrors Gist's 5-way pattern + adds 3 DNS-specific failure modes (servfail, truncated, rate_limited) — every retry-decisionable failure has its own status; nothing collapses to `error`. This is now the load-bearing convention for any future Foxbook adapter (week-2 Tier-3 GitHub Org / DMARC email will use the same shape).
